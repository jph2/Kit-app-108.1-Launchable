# Applying the Latest Changes from GitHub

This guide walks through downloading the updated launchable stack, verifying the files, and starting the containers. It assumes you have a GitHub account and access to the `jph2/Kit-app-108.1-Launchable` repository.

## 1. Clone or Update the Repository

### First-Time Clone
```bash
git clone https://github.com/jph2/Kit-app-108.1-Launchable.git
cd Kit-app-108.1-Launchable
```

### Pulling Updates
If the project folder already exists you can either run the helper script or follow the Git commands manually.

**Option A – use the helper script (recommended for non-developers):**
```bash
cd Kit-app-108.1-Launchable
./update-to-latest.sh
```

**Option B – run the Git commands yourself:**
```bash
cd Kit-app-108.1-Launchable
git fetch origin
git checkout work
git pull --ff-only origin work
```
Both options synchronize your local copy with the latest commits from GitHub (including the refreshed Docker files and scripts).

## 2. Inspect the `kit-app-108` Folder
All runtime files live in the `kit-app-108` subdirectory. Confirm that these files exist:

- `docker-compose.yml`
- `docker-compose.override.yml`
- `start-kit-app.sh`
- `nginx/` (contains proxy configuration)
- `web-viewer-sample/` (Vite-based viewer)

If any file is missing, repeat the `git pull` step to ensure your checkout is up to date.

## 3. Launch the Containers
From inside `kit-app-108` run:
```bash
cd kit-app-108
docker compose up -d
./start-kit-app.sh
```

This sequence starts the Isaac Sim container stack and launches the Kit App streaming process.

## 4. Access the Viewer
Open a browser to:
- `http://<instance-ip>/viewer` for plain HTTP, or
- `https://<instance-ip>/viewer` for the TLS endpoint generated by the proxy.

When running in Brev, replace `<instance-ip>` with the Brev-provided hostname.

## 5. (Optional) Committing Your Own Changes
If you edit any files and want to push them back to GitHub:
```bash
git status
git add <files>
git commit -m "Describe your update"
git push origin <branch-name>
```

For personal experiments, create a new branch: `git checkout -b my-feature` before editing files. This keeps the `work` branch clean.
